## **[[1.2.3 Native Validation & Feedback Semantics]]**

### **Objective**

Build a precise mental and practical model of how browsers perform native form validation, enforce constraints on user input, and surface meaningful feedback before submission.

### **Approach**

Treat validation as both a **declarative contract (HTML constraints)** and a **runtime system (browser-driven checks and UI feedback)**. Study it from the dual perspective of **author-defined rules** and **user-facing error handling**.

### **Method**

- Start with required fields using `required` and observe blocking behavior on submission.
    
- Apply pattern constraints with `pattern` and evaluate regex-based input enforcement.
    
- Enforce numeric and date ranges using `min`, `max`, and `step`.
    
- Combine constraints with semantic input types (`email`, `url`, `number`, `date`) to trigger built-in checks.
    
- Explore the **Constraint Validation API**:  
    `checkValidity()`, `reportValidity()`, `setCustomValidity()`.
    
- Inspect **validity states** via the `validity` object (`valueMissing`, `typeMismatch`, `patternMismatch`, `rangeUnderflow`, `rangeOverflow`, `tooShort`, `tooLong`, `stepMismatch`, `customError`, `valid`).
    
- Observe native error messages and focus behavior when constraints fail.
    
- Test the effect of `novalidate` on the `<form>` and `formnovalidate` on submit buttons.
    

### **Constraints**

- Do not rely on client-side validation alone; always validate on the server.
    
- Avoid disabling native validation unless you fully replace its behavior and feedback.
    
- Do not hide errors without providing accessible alternatives.
    
- Never assume patterns guarantee correct or safe data.
    
- Ensure validation feedback is perceivable by keyboard and screen reader users.
    
- Avoid over-constraining inputs in ways that block legitimate user data.

## **Beginner Level — 3 Questions**

1. What does the `required` attribute do in an HTML form field, and how does the browser respond if it is left empty?
    
2. How is the `pattern` attribute used for input validation? Give a simple example.
    
3. What are the `min` and `max` attributes used for in number or date inputs?
    

---

## **Advanced Level — 4 Questions**

1. What is native form validation in HTML5, and how does it differ from JavaScript-only validation?
    
2. Explain the purpose of the **Constraint Validation API**. Which methods are commonly used?
    
3. What are validity states (such as `valueMissing` or `patternMismatch`), and how can developers access them?
    
4. How do `novalidate` on `<form>` and `formnovalidate` on submit buttons affect validation behavior?
    

---

## **Professional / Interview Level — 5 Questions**

1. How would you design a validation strategy that combines native HTML validation with custom JavaScript logic without breaking accessibility?
    
2. In a real application, how do you provide user-friendly feedback messages while still relying on browser validation mechanisms?
    
3. Describe a scenario where native validation is insufficient. How would you extend it using the Constraint Validation API?
    
4. How do different validity states influence UX, and how can they be mapped to custom error messages and UI states?
    
5. What are the risks of disabling native validation globally, and how would you justify doing so in a production system?
