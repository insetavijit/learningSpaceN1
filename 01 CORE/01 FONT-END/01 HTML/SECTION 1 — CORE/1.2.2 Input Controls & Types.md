## **[[1.2.2 Input Controls & Types]]**

### **Objective**

Build a precise mental and practical model of how HTML input controls capture user data, enforce basic constraints, and expose consistent value semantics for form submission and scripting.

### **Approach**

Treat each control as both a **user interface widget (UX interaction)** and a **data source (name → value pair)**. Study controls from the dual perspective of **usability** and **browser-native validation and value handling**.

### **Method**

- Start with the anatomy of the `<input>` element and common types:  
    `text`, `email`, `password`, `number`, `date`, `checkbox`, `radio`, `file`, `hidden`, `submit`, `reset`.
    
- Practice choosing the correct control for each data type instead of relying on generic text inputs.
    
- Use `<textarea>` for multi-line free text and observe differences from text inputs.
    
- Build choice inputs using `<select>` and `<option>` and compare with radio and checkbox groups.
    
- Use `<button>` with explicit `type` values (`submit`, `button`, `reset`) to control form behavior.
    
- Experiment with native attributes:  
    `required`, `min`, `max`, `step`, `maxlength`, `pattern`, `placeholder`, `readonly`, `disabled`.
    
- Inspect how different input types trigger **native widgets** (date pickers, number steppers, color pickers, file dialogs) across browsers.
    
- Observe how values are normalized and serialized for each control during submission.
    

### **Constraints**

- Always provide a `<label>` for every control.
    
- Do not misuse input types; prefer semantic types over `type="text"`.
    
- Avoid relying on `placeholder` as a replacement for labels.
    
- Do not leave the default `<button>` type ambiguous; always declare it.
    
- Avoid disabling controls unless they are truly inactive, as disabled fields do not submit values.
    
- Do not replace native widgets unless there is a clear UX or accessibility benefit.

## **Beginner Level — 3 Questions**

1. What is the `<input>` element, and name a few common input types used in HTML forms.
    
2. What is the difference between `<input type="text">` and `<textarea>`?
    
3. How do `<select>` and `<option>` elements work together to create a dropdown list?
    

---

## **Advanced Level — 4 Questions**

1. Explain the purpose of different `<button>` types (`submit`, `reset`, `button`). How do they affect form behavior?
    
2. What are native input widgets, and how do types like `email`, `date`, and `number` improve user experience?
    
3. Compare `<input type="radio">` and `<input type="checkbox">`. How is grouping achieved for each?
    
4. How do attributes like `name`, `value`, and `disabled` influence form data submission?
    

---

## **Professional / Interview Level — 5 Questions**

1. How do modern input types contribute to both **client-side validation** and **mobile-friendly UI**, and what are their limitations?
    
2. In a large production form, how would you choose between native controls and custom-styled components while preserving accessibility?
    
3. Describe how browsers serialize different input controls (text, checkbox, radio, select, textarea) into a submission payload.
    
4. What issues can arise when relying solely on native widgets across different browsers and devices, and how would you mitigate them?
    
5. How would you design a robust input strategy for collecting international phone numbers, dates, and currencies using HTML input types?
