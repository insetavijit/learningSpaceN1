## **[[1.3.1 Native Accessibility Semantics]]**

### **Objective**

Build a precise mental and practical model of how native HTML semantics automatically expose accessibility information to assistive technologies without requiring explicit ARIA.

### **Approach**

Treat native elements as both **visual structure** and **accessibility APIs inputs**. Study them from the dual perspective of **semantic markup** and **how browsers map it into the accessibility tree**.

### **Method**

- Identify **implicit ARIA roles** provided by native elements (e.g., `<button>` → `button`, `<nav>` → `navigation`, `<main>` → `main`).
    
- Map **landmark roles** created by elements such as `<header>`, `<nav>`, `<main>`, `<aside>`, and `<footer>` and observe quick-jump navigation in screen readers.
    
- Study **interactive element semantics** for controls like `<a>`, `<button>`, `<input>`, `<select>`, and how their roles and states are exposed.
    
- Observe **focusability rules**: which elements are focusable by default and how `tabindex="0"` and `-1` affect keyboard flow.
    
- Inspect the accessibility tree in DevTools to verify roles, names, and states derived from markup.
    
- Compare native semantics with ARIA overrides to understand when ARIA is unnecessary or harmful.
    
- Test keyboard and screen reader interaction to confirm expected behavior.
    

### **Constraints**

- Prefer native HTML semantics over ARIA whenever possible.
    
- Do not override implicit roles unless there is a strong, justified need.
    
- Avoid adding ARIA roles that duplicate native semantics.
    
- Never change semantics in ways that break expected keyboard or screen reader behavior.
    
- Do not make non-interactive elements focusable without providing proper interaction logic.
    
- Avoid hiding meaningful content from the accessibility tree.

|---|---|

|**Landmark roles**|High-level structural roles conveyed by elements like `<header>`, `<nav>`, `<main>`, `<aside>`, and `<footer>`, enabling assistive technologies to offer region-based navigation of a page.|
|**Interactive element semantics**|The default behaviors and states of controls such as links, buttons, inputs, and menus, which communicate purpose, operability, and state to users and assistive technologies.|
|**Focusability rules**|The browser-defined rules that determine which elements can receive focus, in what order, and under which conditions, governing keyboard navigation and interaction flow.|

## **Advanced Level — 4 Questions**

2. Why is it usually discouraged to override native semantics with explicit ARIA roles?
    
3. How do interactive elements like `<button>`, `<a>`, and form controls differ in their default accessibility behavior?
